这篇用于沉淀金蝶 BOS/苍穹二开的实战方法。目标不是只知道“能写插件”，而是知道什么时候该配置、什么时候该扩展、什么时候该写插件，以及如何保证二开不破坏业务和财务闭环。

## 一、二开前先判断要不要开发

需求处理优先级：

```text
标准功能
  -> 参数配置
  -> 单据扩展
  -> 工作流/权限配置
  -> 报表配置
  -> 轻量插件
  -> 服务接口
  -> 深度定制
```

不要一上来就写代码。开发会带来测试、发布、升级、运维和责任边界。

| 需求 | 优先方案 |
| --- | --- |
| 新增字段 | 单据扩展或基础资料扩展 |
| 新增审批节点 | 工作流配置 |
| 新增校验 | 单据校验规则或操作插件 |
| 自动带出字段 | 表单规则、基础资料属性、插件 |
| 单据转换 | 单据转换规则 |
| 外部系统同步 | OpenAPI、集成方案 |
| 大屏或统计 | 报表、数据集、BI |
| 特殊业务算法 | 服务插件或后台服务 |

## 二、开发对象地图

| 对象 | 说明 | 常见场景 |
| --- | --- | --- |
| 应用 | 业务域或模块 | 财务扩展、供应链扩展 |
| 基础资料 | 相对稳定的主数据 | 客户扩展字段、物料属性 |
| 单据 | 业务交易数据 | 费用申请、业务申请单 |
| 动态表单 | 不落标准业务单据的交互界面 | 查询、向导、批处理 |
| 列表 | 单据列表展示和筛选 | 增加按钮、批量操作 |
| 操作 | 保存、提交、审核、反审核等动作 | 增加校验和后置处理 |
| 工作流 | 审批流程 | 金额分级审批 |
| 权限 | 菜单、按钮、数据范围 | 岗位职责分离 |
| 插件 | Java 或脚本扩展逻辑 | 复杂校验、接口调用 |

## 三、建模方法

### 单据设计步骤

1. 确定业务目的。
2. 确定单据是否需要编码规则。
3. 设计表头字段。
4. 设计分录字段。
5. 确定状态流：暂存、提交、审核、关闭。
6. 确定操作按钮。
7. 确定工作流。
8. 确定单据转换关系。
9. 确定权限。
10. 确定是否生成凭证或影响财务。

### 字段设计原则

| 字段类型 | 用法 |
| --- | --- |
| 基础资料字段 | 引用客户、供应商、物料、部门等 |
| 枚举字段 | 状态、类型、业务分类 |
| 金额字段 | 必须明确币别、税额、含税/不含税 |
| 数量字段 | 要明确单位和精度 |
| 日期字段 | 区分业务日期、申请日期、审核日期 |
| 文本字段 | 摘要、说明、外部编号 |
| 附件字段 | 合同、发票、回单、图片 |

字段命名要稳定，避免上线后反复改字段标识。字段一旦被接口、报表、插件、凭证模板使用，改名会牵一发动全身。

## 四、单据生命周期

```text
新增
  -> 保存
  -> 提交
  -> 审批
  -> 审核
  -> 下推或生成下游单据
  -> 关闭
  -> 反审核或作废
```

每个阶段适合做的事不同：

| 阶段 | 适合处理 |
| --- | --- |
| 保存前 | 字段完整性、金额数量基础校验 |
| 保存后 | 自动生成辅助记录、日志 |
| 提交前 | 是否满足提交条件 |
| 审核前 | 核心业务规则校验 |
| 审核后 | 生成下游单据、通知、接口同步 |
| 反审核前 | 检查是否已被下游引用 |
| 删除前 | 检查是否允许删除 |

## 五、插件开发思路

### 常见插件类型

| 插件 | 适用场景 |
| --- | --- |
| 表单插件 | 页面交互、字段联动、按钮控制 |
| 操作插件 | 保存、提交、审核等操作前后处理 |
| 服务插件 | 后台业务逻辑、批量处理 |
| 列表插件 | 列表按钮、批量操作、过滤 |
| 单据转换插件 | 上游到下游字段映射和特殊逻辑 |
| 定时任务 | 定时同步、定时检查 |

### 插件设计模板

写插件前先写清：

```text
业务场景：
触发时机：
输入数据：
输出数据：
影响单据：
影响财务：
异常处理：
权限要求：
日志要求：
测试用例：
```

### 插件开发注意事项

- 不要在保存事件里做很慢的外部接口调用。
- 不要绕过标准审核和反审核逻辑直接改数据。
- 不要在生产环境直接写库改核心单据。
- 金额、税额、币别、汇率要用统一精度。
- 插件失败要给用户可理解的错误信息。
- 关键操作要写日志。
- 批量处理要考虑性能和事务。

## 六、单据转换

单据转换是金蝶业务链路的关键，例如销售订单下推销售出库，应收单生成凭证。

设计单据转换要确认：

- 上游单据和下游单据。
- 表头字段如何映射。
- 分录字段如何映射。
- 数量如何控制。
- 是否允许超额下推。
- 是否支持部分下推。
- 下游反审核是否影响上游。
- 是否需要携带源单关联。

源单关联非常重要。没有源单关联，后续追溯、对账、反写数量、接口回查都会困难。

## 七、权限和工作流

### 权限设计

| 权限 | 说明 |
| --- | --- |
| 菜单权限 | 是否能进入功能 |
| 按钮权限 | 是否能新增、审核、删除、导出 |
| 数据权限 | 能看哪些组织、部门、客户、供应商 |
| 字段权限 | 敏感字段是否可见或可编辑 |
| 流程权限 | 审批节点能否处理 |

### 工作流设计

工作流要回答：

- 谁发起。
- 谁审批。
- 金额或业务类型是否分级。
- 审批人缺席怎么办。
- 驳回回到哪一步。
- 审批通过后是否自动审核。
- 审批记录是否可追溯。

## 八、财务相关二开红线

涉及财务的二开尤其要谨慎。

不要做：

- 直接修改已过账凭证。
- 绕过应收应付直接改总账余额。
- 直接更新核销结果。
- 删除已月结期间单据。
- 改金额但不留痕。
- 外部系统重复推单没有幂等控制。

必须做：

- 保留外部单号。
- 保留源单关联。
- 金额税额有校验。
- 凭证生成能追溯。
- 反审核检查下游。
- 日志记录关键字段。

## 九、测试清单

### 功能测试

- [ ] 新增、保存、提交、审核、反审核可用。
- [ ] 必填字段校验正确。
- [ ] 字段联动正确。
- [ ] 单据转换正确。
- [ ] 权限控制正确。
- [ ] 工作流正确。
- [ ] 异常提示可理解。

### 数据测试

- [ ] 金额精度正确。
- [ ] 数量单位换算正确。
- [ ] 币别汇率正确。
- [ ] 源单关联正确。
- [ ] 报表能取到字段。
- [ ] 接口能识别唯一键。

### 财务回归

- [ ] 不影响凭证模板。
- [ ] 不影响应收应付核销。
- [ ] 不影响总账过账。
- [ ] 不影响月结。
- [ ] 不影响报表公式。

## 十、上线和发布

二开上线前要准备：

- 需求说明。
- 设计说明。
- 代码或配置包。
- 测试报告。
- 发布步骤。
- 回滚方案。
- 影响范围说明。
- 用户操作说明。

上线后至少观察：

- 单据是否能正常保存审核。
- 接口是否报错。
- 日志是否正常。
- 凭证生成是否正常。
- 月结是否受影响。

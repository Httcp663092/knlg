单据转换、工作流和权限是金蝶二开与实施中最容易相互影响的三块。一个字段扩展可能影响下推，一个审批节点可能影响审核，一个权限设置可能导致用户看不到下游单据。做二开时必须把这三块一起设计和测试。

## 一、三者关系

```text
权限
  -> 决定谁能看、谁能点、谁能处理数据

工作流
  -> 决定单据提交后谁审批、审批通过后如何生效

单据转换
  -> 决定上游单据如何生成下游单据和反写执行情况
```

三者共同决定业务链路是否闭环。

## 二、单据转换设计

### 转换设计要素

| 要素 | 说明 |
| --- | --- |
| 源单 | 上游单据，例如销售订单 |
| 目标单 | 下游单据，例如销售出库 |
| 转换路线 | 从哪个操作触发 |
| 字段映射 | 表头和分录字段如何带入 |
| 数量控制 | 是否允许超额、部分、重复下推 |
| 关联关系 | 是否保留源单和源分录 |
| 反写规则 | 下游审核后如何更新上游执行数量 |
| 关闭规则 | 上游何时关闭 |

### 字段映射

字段映射要区分：

- 直接带入。
- 按基础资料带出。
- 按公式计算。
- 按组织或单据类型取默认值。
- 下游手工维护。

涉及金额、税额、币别、汇率时，要明确精度和来源。

### 数量控制

常见规则：

| 规则 | 说明 |
| --- | --- |
| 不允许超额 | 下游数量不能超过上游未执行数量 |
| 允许合理超额 | 收货允许少量超收 |
| 部分下推 | 上游一行可分多次下推 |
| 合并下推 | 多张源单合并为一张目标单 |
| 拆分下推 | 一张源单分录拆成多张目标单 |

## 三、单据转换测试

必须测试：

- 正常下推。
- 部分下推。
- 超额控制。
- 多源单合并。
- 下游反审核。
- 下游删除或作废。
- 上游关闭。
- 权限不足时下推。
- 字段为空时下推。

## 四、工作流设计

### 审批流要素

| 要素 | 示例 |
| --- | --- |
| 发起条件 | 单据提交 |
| 审批人 | 部门负责人、金额负责人 |
| 审批条件 | 金额、组织、单据类型 |
| 驳回规则 | 回到申请人或上一节点 |
| 自动通过 | 小额或同部门可自动通过 |
| 超时处理 | 自动提醒或升级 |
| 审批后动作 | 自动审核、发送通知 |

### 审批流和单据状态

常见状态关系：

```text
暂存
  -> 提交
  -> 审批中
  -> 审批通过
  -> 审核
```

项目中要确认：

- 审批通过是否等于审核。
- 驳回后单据状态是什么。
- 反审核是否需要走审批。
- 修改已审批单据是否重新审批。

## 五、权限扩展

### 权限维度

| 权限 | 说明 |
| --- | --- |
| 菜单 | 是否能进入功能 |
| 按钮 | 新增、保存、提交、审核、删除、导出 |
| 数据 | 组织、部门、客户、供应商、仓库 |
| 字段 | 敏感字段显示或编辑 |
| 工作流 | 是否能审批 |
| 报表 | 能看哪些统计结果 |

### 扩展字段权限

新增字段后要确认：

- 哪些角色能看。
- 哪些角色能编辑。
- 列表是否展示。
- 导出是否包含。
- 接口是否传输。
- 报表是否取数。

## 六、三者联动案例

### 场景：采购申请下推采购订单

业务要求：

- 需求部门提交采购申请。
- 部门经理审批。
- 采购员下推采购订单。
- 采购经理审核订单。
- 申请人只能看到本部门申请。

设计：

- 采购申请新增部门和申请人字段。
- 工作流按部门经理审批。
- 单据转换把申请物料、数量、需求日期带到采购订单。
- 下推时采购员可调整供应商和价格。
- 数据权限限制申请人只能看本部门。

测试：

1. 申请人提交。
2. 部门经理审批。
3. 非采购员尝试下推，应该失败。
4. 采购员下推订单。
5. 采购订单反写申请已下推数量。
6. 采购订单反审核后申请执行数量恢复。

## 七、常见问题

| 问题 | 原因 | 处理 |
| --- | --- | --- |
| 下推后字段为空 | 字段映射漏配或源字段为空 | 补映射和默认值 |
| 下游不能反审核 | 已生成更下游单据 | 先处理下游 |
| 审批人为空 | 岗位、部门、组织映射错误 | 检查人员和岗位 |
| 用户能看菜单但不能审核 | 按钮权限或审批任务权限不足 | 补按钮和流程权限 |
| 报表查不到扩展字段 | 报表数据源未更新 | 更新报表取数字段 |

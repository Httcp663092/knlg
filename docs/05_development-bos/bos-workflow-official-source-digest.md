## 二、BOS/苍穹开发的定位

官方资料把苍穹和开发平台定位为支撑业务对象建模、页面建模、服务扩展、流程扩展、移动能力和开放集成的平台。项目里可以简单理解：

```text
标准应用不够
  -> 先配置参数和流程
  -> 再扩展字段和单据
  -> 再做单据转换和报表
  -> 最后才写插件或接口
```

这和本知识库里“不要一上来就写代码”的原则一致。

## 三、单据建模：先判断对象类型

官方单据建模资料强调单据适合保存业务流水数据，并会自动带出表单、列表、移动表单、移动列表等页面。项目里做建模前要先判断：

| 类型 | 适合场景 |
| --- | --- |
| 单据 | 采购申请、费用申请、质量处理、维修工单 |
| 基础资料 | 供应商等级、设备、工装、费用项目 |
| 动态表单 | 临时查询、向导、批处理界面 |
| 报表 | 管理分析、对账、汇总查询 |

不要把主数据做成普通单据，也不要把业务流水做成基础资料。

## 四、基础资料建模：组织和引用最关键

基础资料官方说明里强调它可以被基础资料字段引用，并且可选择带组织或不带组织模板。项目中要重点判断：

- 是否全集团共享。
- 是否按组织隔离。
- 是否需要上级下级层级。
- 是否会被业务单据大量引用。
- 是否要分配到其他组织。

### 典型判断

| 数据 | 建议 |
| --- | --- |
| 费用项目 | 基础资料，通常全局共享或按组织分配 |
| 设备台账 | 基础资料，可按组织 |
| 质量缺陷类型 | 基础资料，通常全局共享 |
| 项目资料 | 基础资料，可能按组织或项目组织 |
| 临时处理记录 | 单据 |

## 五、业务流程和单据转换

官方业务流程资料强调单据之间的流转、下推、关联和转换。项目里最关键的是保留源单关系。

```text
源单
  -> 单据转换
  -> 目标单
  -> 反写执行数量
  -> 关闭或继续执行
```

### 设计要点

- 源单和目标单是什么。
- 是否允许部分下推。
- 是否允许超额下推。
- 字段如何映射。
- 下游审核后如何反写。
- 反审核下游是否恢复上游数量。
- 删除或作废下游如何处理。

### 常见坑

| 问题 | 后果 |
| --- | --- |
| 不保留源单关联 | 无法联查、无法反写、对账困难 |
| 字段映射漏配 | 下游单据字段为空 |
| 超额控制不清 | 采购、发货、领料数量超业务口径 |
| 反审核规则没测 | 上下游状态错乱 |

## 六、流程设计器和工作流

官方流程设计器和工作流资料说明系统可按业务规则配置审批路线。项目里要把工作流理解成“内控证据”，不是单纯的审批按钮。

### 流程设计要素

- 发起条件。
- 审批节点。
- 审批人来源。
- 金额、部门、组织条件。
- 驳回规则。
- 超时提醒。
- 审批通过后的动作。
- 审批记录保留。

### 常见审批人来源

- 固定人员。
- 部门负责人。
- 岗位。
- 角色。
- 申请人上级。
- 按金额或组织匹配。

### 测试场景

- 正常审批。
- 驳回后修改再提交。
- 审批人为空。
- 审批人离职。
- 超金额审批。
- 跨组织审批。
- 移动端审批。

## 七、BOS 参数设置

BOS 参数通常影响系统底层行为，例如编码、查询、基础资料、附件、权限或性能。项目里不能随意改参数。

### 参数变更流程

```text
提出变更
  -> 说明业务原因
  -> 测试环境验证
  -> 影响范围评估
  -> 生产变更
  -> 变更后回归
  -> 记录归档
```

### 参数变更风险

- 影响所有用户。
- 影响历史单据展示。
- 影响接口或二开。
- 影响报表取数。
- 影响权限或基础资料引用。

## 八、开发交付检查清单

- [ ] 标准功能、参数、流程、扩展、插件的边界已判断。
- [ ] 业务对象类型选择正确。
- [ ] 基础资料组织隔离规则已确认。
- [ ] 单据转换保留源单和分录关联。
- [ ] 工作流覆盖正常、驳回、超额、审批人为空等场景。
- [ ] 参数变更有测试和回滚说明。
- [ ] 字段扩展已回归接口、报表、凭证模板和权限。
- [ ] 插件开发有日志、幂等和异常提示。

科目和核算维度是金蝶财务系统的骨架。科目决定账务分类，核算维度决定管理分析口径。很多财务系统上线后报表不准、凭证生成失败、辅助余额混乱、月结困难，本质都是科目和维度设计不清。

## 一、科目和维度的区别

| 项目 | 科目 | 核算维度 |
| --- | --- | --- |
| 作用 | 反映会计要素和经济业务类别 | 反映业务对象、责任对象和管理分析口径 |
| 示例 | 应收账款、主营业务收入、管理费用 | 客户、供应商、部门、项目、员工、物料 |
| 报表 | 资产负债表、利润表、科目余额表 | 客户余额、供应商余额、部门费用、项目利润 |
| 设计原则 | 稳定、合规、层级清晰 | 能支撑管理分析，避免过多 |

简单理解：

```text
科目回答：这是什么经济业务？
维度回答：和谁有关、归属哪里、为哪个项目发生？
```

## 二、科目体系设计

### 科目层级

科目通常按会计要素分为：

- 资产类。
- 负债类。
- 共同类。
- 所有者权益类。
- 成本类。
- 损益类。

示例：

```text
1122 应收账款
  1122.01 应收货款
  1122.02 应收服务款

6602 管理费用
  6602.01 办公费
  6602.02 差旅费
  6602.03 折旧费
```

科目层级不要无限细化。太细会导致凭证模板复杂、报表维护困难、人员录入容易错。

### 科目细化还是维度细化

一个常见问题：到底新增科目，还是用核算维度？

| 需求 | 建议 |
| --- | --- |
| 按费用类型区分 | 用科目，例如差旅费、办公费 |
| 按部门看费用 | 用部门维度 |
| 按客户看应收 | 用客户维度 |
| 按项目看利润 | 用项目维度 |
| 按产品线看收入 | 可以用维度或物料分类 |
| 按地区看销售 | 用客户地区或销售组织维度 |

原则：

- 会计性质不同，用科目。
- 管理对象不同，用维度。
- 法定报表需要稳定科目，管理报表更适合维度。

## 三、核算维度设计

常用维度：

| 维度 | 适用科目 |
| --- | --- |
| 客户 | 应收账款、预收账款、主营业务收入 |
| 供应商 | 应付账款、预付账款、采购暂估 |
| 部门 | 管理费用、销售费用、制造费用 |
| 员工 | 备用金、其他应收款、报销 |
| 项目 | 项目收入、项目成本、项目费用 |
| 物料 | 存货、收入、成本分析 |
| 资产类别 | 固定资产、累计折旧 |
| 银行账户 | 银行存款 |
| 组织 | 多组织核算和汇总 |

维度不是越多越好。每加一个必录维度，都会增加录入、接口、凭证模板、报表、对账成本。

## 四、金蝶中科目和维度的落地

在金蝶里要关注：

- 科目表。
- 科目级次。
- 科目余额方向。
- 是否数量金额核算。
- 是否外币核算。
- 是否启用核算维度。
- 维度是否必录。
- 维度取值来源。
- 凭证模板能否带出维度。
- 报表公式是否包含新科目。

## 五、科目启用辅助核算的影响

如果科目启用了客户辅助核算，那么凭证分录必须带客户。

例如：

```text
借：应收账款 / 客户A 113,000
  贷：主营业务收入 100,000
  贷：应交税费-销项税 13,000
```

如果应收账款启用了客户维度，但凭证模板取不到客户，就会出现凭证生成失败或辅助核算为空。

## 六、典型科目维度设计

### 应收账款

建议维度：

- 客户。
- 业务组织或核算组织。
- 币别。

可选维度：

- 销售部门。
- 销售员。
- 项目。

不要把每个客户都建成明细科目。客户应作为核算维度，否则客户多时科目表会失控。

### 应付账款

建议维度：

- 供应商。
- 采购组织或核算组织。
- 币别。

可选维度：

- 采购部门。
- 项目。

### 费用类科目

建议：

- 科目区分费用类型。
- 部门维度区分责任部门。
- 项目维度区分专项项目。
- 员工维度用于报销或备用金类科目。

示例：

```text
借：管理费用-差旅费 / 部门=销售部 / 员工=张三
  贷：其他应付款 / 员工=张三
```

### 存货类科目

常见科目：

- 原材料。
- 半成品。
- 库存商品。
- 委托加工物资。
- 发出商品。
- 存货跌价准备。

维度可按：

- 物料。
- 仓库。
- 库存组织。
- 成本中心。

但存货详细数量和批号通常由库存和存货核算模块管理，总账不一定需要把所有明细维度都带上。

## 七、科目和业务模块的关系

| 业务模块 | 常见科目 | 维度来源 |
| --- | --- | --- |
| 销售应收 | 应收账款、主营业务收入、销项税 | 客户、销售部门、项目 |
| 采购应付 | 原材料、应付账款、进项税、暂估 | 供应商、物料、仓库 |
| 库存核算 | 原材料、库存商品、主营业务成本 | 物料、库存组织、仓库 |
| 固定资产 | 固定资产、累计折旧、折旧费用 | 资产类别、部门 |
| 费用报销 | 管理费用、销售费用、其他应付款 | 部门、员工、项目 |
| 出纳资金 | 银行存款、现金 | 银行账户、币别 |

## 八、凭证模板中的维度来源

凭证模板要明确每个维度从哪里来：

| 维度 | 常见来源 |
| --- | --- |
| 客户 | 应收单客户、销售订单客户 |
| 供应商 | 应付单供应商、采购订单供应商 |
| 部门 | 单据部门、费用承担部门、人员部门 |
| 项目 | 业务单据项目字段 |
| 物料 | 单据分录物料 |
| 员工 | 报销人、业务员、经办人 |
| 银行账户 | 收付款账户 |

如果源单没有字段，模板就取不到维度。不要上线后才发现报表无法按项目统计。

## 九、科目变更管理

科目一旦启用，修改要谨慎。

常见变更：

- 新增科目。
- 禁用科目。
- 修改辅助核算。
- 修改余额方向。
- 调整报表公式。
- 调整凭证模板。

变更步骤：

1. 财务提出需求。
2. 判断是否必须新增科目，还是用维度解决。
3. 测试环境配置。
4. 调整凭证模板和报表公式。
5. 用典型业务单据生成凭证测试。
6. 检查科目余额表和财务报表。
7. 生产变更并记录。

## 十、常见问题

| 问题 | 原因 | 处理 |
| --- | --- | --- |
| 凭证生成提示辅助核算为空 | 科目启用维度但模板取不到 | 补源单字段或模板取值 |
| 科目余额表有余额但辅助余额为空 | 手工凭证未录维度 | 补录或调整凭证 |
| 报表少了一块金额 | 新增科目未加报表公式 | 更新报表公式 |
| 费用无法按部门分析 | 费用科目未启用部门维度 | 评估后启用维度并规范模板 |
| 客户太多导致科目表混乱 | 把客户建成明细科目 | 改用客户辅助核算 |

## 十一、实际案例：应收按客户和项目双维度核算

背景：

- 企业做项目制销售。
- 同一客户可能有多个项目。
- 财务希望既能看客户欠款，又能看项目回款。

设计：

- 应收账款科目启用客户和项目维度。
- 销售订单和应收单必须录入项目。
- 凭证模板从应收单带客户和项目。
- 应收账龄按客户查。
- 项目回款报表按项目查。

风险：

- 如果销售订单漏项目，应收凭证无法完整带出。
- 如果手工凭证没有项目，会导致项目余额不准。

## 十二、设计检查清单

- [ ] 科目表层级清晰，明细不过度。
- [ ] 客户、供应商使用维度而不是大量明细科目。
- [ ] 费用类科目和部门维度分工清楚。
- [ ] 项目、物料、员工等维度有明确用途。
- [ ] 凭证模板能带出所有必录维度。
- [ ] 报表公式包含新增科目。
- [ ] 手工凭证录入规则已培训。
- [ ] 科目和维度变更有审批和测试。

这篇用于把金蝶项目从“能开会”推进到“能上线、能验收、能运维”。实施不是简单配置系统，而是把客户业务、系统功能、数据口径、岗位职责和上线风险组织成一套可落地方案。

## 一、项目阶段总览

```text
项目启动
  -> 现状调研
  -> 蓝图设计
  -> 系统配置
  -> 数据准备
  -> 集成和二开
  -> 测试验证
  -> 培训
  -> 上线切换
  -> 稳定支持
  -> 验收归档
```

| 阶段 | 目标 | 核心交付物 |
| --- | --- | --- |
| 启动 | 明确范围、组织和计划 | 项目章程、计划、通讯录、风险清单 |
| 调研 | 搞清当前业务和痛点 | 调研纪要、流程图、问题清单 |
| 蓝图 | 确定未来业务和系统方案 | 蓝图文档、差异清单、配置清单 |
| 配置 | 按蓝图搭建系统 | 参数、基础资料、流程、权限 |
| 数据 | 准备主数据和期初 | 导入模板、校验报告、签字确认 |
| 测试 | 证明系统能用 | 测试用例、缺陷清单、测试报告 |
| 培训 | 让用户会用 | 培训材料、签到表、操作手册 |
| 上线 | 完成切换 | 上线计划、切换清单、应急方案 |
| 支持 | 解决上线初期问题 | 问题台账、补丁清单、优化建议 |
| 验收 | 固化成果 | 验收报告、运维移交、项目归档 |

## 二、项目启动

启动会要解决：

- 项目范围：上线哪些模块，哪些暂不做。
- 组织范围：哪些公司、工厂、仓库、部门。
- 时间计划：调研、蓝图、测试、上线节点。
- 角色分工：客户项目经理、关键用户、实施顾问、开发、运维。
- 决策机制：需求变更、风险升级、验收标准。

启动会输出：

- 项目计划。
- 组织通讯录。
- 会议机制。
- 风险清单。
- 资料收集清单。

## 三、调研方法

### 调研原则

调研不要只问“你们现在怎么操作”，还要问：

1. 为什么这么做。
2. 这个动作的输入是什么。
3. 输出给谁用。
4. 出错后谁负责。
5. 月末怎么对账。
6. 哪些口径是老板、财务、业务都认可的。

### 财务调研

重点问题：

- 核算主体有哪些。
- 是否多账簿、多准则、多币别。
- 科目表是否沿用旧系统。
- 应收、应付、固定资产、存货期初怎么来。
- 月结顺序和责任人。
- 报表有哪些，取数口径是什么。
- 哪些凭证由业务生成，哪些手工录入。

### 供应链调研

重点问题：

- 采购从申请到付款的流程。
- 销售从订单到收款的流程。
- 仓库、库位、批号、序列号管理要求。
- 采购、销售、库存是否存在跨组织业务。
- 存货计价方式。
- 盘点频率和盘盈盘亏处理。

### 制造调研

重点问题：

- 产品 BOM 和工艺路线是否完整。
- MRP 是否使用。
- 生产订单如何下达。
- 领料方式是按单、配套还是倒冲。
- 是否委外。
- 成本核算对象和费用分配规则。

### 开发集成调研

重点问题：

- 哪些需求标准功能支持，哪些需要二开。
- 外部系统有哪些。
- 接口方向是进金蝶还是出金蝶。
- 是否需要实时同步。
- 是否有历史数据迁移。
- 权限、日志、审计要求。

## 四、蓝图设计

蓝图要把“以后怎么做”写清楚。

### 蓝图结构

```text
业务范围
  -> 组织架构
  -> 主数据方案
  -> 业务流程
  -> 系统配置
  -> 单据和审批
  -> 凭证和报表
  -> 权限和岗位
  -> 数据迁移
  -> 二开和接口
  -> 风险和待确认事项
```

### 蓝图评审

蓝图评审必须有客户关键用户参加。评审时不要只展示流程图，要拿典型业务走一遍：

- 一笔采购怎么从申请到付款。
- 一笔销售怎么从订单到收款。
- 一笔生产怎么从计划到完工入库。
- 月末怎么结账。
- 报表怎么出。

## 五、数据迁移

### 数据分类

| 数据 | 示例 | 注意 |
| --- | --- | --- |
| 主数据 | 物料、客户、供应商、部门、员工、仓库 | 去重、编码、状态 |
| 业务期初 | 应收明细、应付明细、库存期初、资产卡片 | 要和总账对账 |
| 财务期初 | 科目余额、辅助核算余额、外币余额 | 借贷平衡 |
| 历史数据 | 历史订单、历史凭证、历史报表 | 一般只迁移必要查询 |
| 配置数据 | 科目、核算维度、审批流、权限 | 要和蓝图一致 |

### 数据迁移步骤

1. 发模板。
2. 客户填报。
3. 顾问校验。
4. 试导入。
5. 业务用户校验。
6. 修正数据。
7. 正式导入。
8. 出导入结果。
9. 客户签字确认。

### 数据校验规则

- 必填字段不能为空。
- 编码不能重复。
- 引用资料必须存在。
- 金额类数据要和旧系统汇总一致。
- 应收应付明细要等于总账余额。
- 库存数量金额要和仓库盘点一致。
- 固定资产卡片要和总账资产科目一致。

## 六、测试管理

### 测试类型

| 测试 | 目的 |
| --- | --- |
| 单元测试 | 单个功能是否可用 |
| 集成测试 | 模块之间是否能串起来 |
| UAT | 关键用户按真实业务验收 |
| 性能测试 | 大数据量下是否可用 |
| 权限测试 | 用户是否只能看到该看的数据 |
| 月结演练 | 能否完整关账和出报表 |

### 测试用例格式

| 字段 | 示例 |
| --- | --- |
| 用例编号 | FIN-GL-001 |
| 模块 | 总账 |
| 场景 | 手工凭证录入审核过账 |
| 前置条件 | 账簿、科目、期间已配置 |
| 操作步骤 | 录入凭证、审核、过账 |
| 预期结果 | 科目余额表体现发生额 |
| 实际结果 | 通过或失败 |
| 问题编号 | BUG-001 |

## 七、上线切换

### 上线前冻结

上线前要冻结：

- 主数据新增和修改。
- 旧系统业务录入。
- 期初余额。
- 库存盘点结果。
- 未结业务清单。

### 切换清单

- [ ] 生产环境版本确认。
- [ ] 用户和权限确认。
- [ ] 主数据导入完成。
- [ ] 期初数据导入完成。
- [ ] 期初对账完成。
- [ ] 关键流程测试通过。
- [ ] 接口联调通过。
- [ ] 用户培训完成。
- [ ] 应急联系人确认。
- [ ] 回退方案确认。

### 上线当天安排

| 时间 | 工作 |
| --- | --- |
| T-1 | 旧系统数据冻结，导出最终数据 |
| T | 正式导入主数据和期初 |
| T+1 | 核对期初，开放业务用户 |
| T+2 | 跟踪首批业务单据 |
| T+3 | 输出上线问题清单 |

## 八、验收

验收不等于系统能登录。验收要看项目目标是否达成。

验收材料：

- 蓝图确认版。
- 配置清单。
- 数据导入和校验报告。
- 测试报告。
- 培训记录。
- 上线切换报告。
- 问题清单及处理结果。
- 操作手册。
- 运维移交文档。

## 九、风险清单

| 风险 | 预防 |
| --- | --- |
| 客户关键用户不参与 | 启动会明确职责和签字节点 |
| 主数据质量差 | 早发模板，早校验，多轮清洗 |
| 需求失控 | 建需求变更流程 |
| 期初对不上 | 先定义对账口径再导入 |
| 二开影响标准流程 | 二开必须测试凭证、报表和权限 |
| 上线后不会用 | 培训加操作手册加现场支持 |
| 月结失败 | 上线前至少做一次月结演练 |

## 十一、实施顾问日常工作法

- 每天维护问题清单。
- 每次会议输出纪要和待办。
- 每个配置变更留记录。
- 每个二开需求写清边界和验收。
- 每次数据导入保留原始文件和导入结果。
- 每个模块至少跑一条端到端业务。
- 月结场景必须演练。

项目调研和蓝图设计是金蝶项目成败的第一道关口。调研不是简单问用户“你现在怎么做”，蓝图也不是把现状照搬进系统；真正有用的蓝图要把业务目标、组织边界、单据流、数据口径、权限职责、接口二开和上线风险全部说清楚。

## 一、适用范围

适用于金蝶云星空、金蝶云苍穹相关项目中的：

- 财务、供应链、生产制造、成本、资产、资金等模块实施。
- 老系统切换到金蝶。
- 多组织、多工厂、多仓库、多账簿项目。
- 存在接口、二开、审批流、报表重构的项目。
- 需要形成正式蓝图和验收依据的项目。

## 二、调研前准备

### 资料清单

调研前先让客户准备资料，不要到现场才临时问。

| 资料 | 用途 | 常见问题 |
| --- | --- | --- |
| 组织架构图 | 确定核算组织、业务组织、库存组织、工厂 | 部门图不等于系统组织模型 |
| 岗位职责 | 设计权限、审批流、操作手册 | 岗位名称相同但职责不同 |
| 现有流程图 | 识别现状动作和痛点 | 很多客户只有口头流程 |
| 单据样张 | 确定字段、审批、打印、接口 | 纸质单据字段不一定需要进系统 |
| 编码规则 | 物料、客户、供应商、仓库、项目等编码 | 编码含义过多会影响维护 |
| 历史报表 | 确定管理口径和取数来源 | 报表口径经常无人能解释 |
| 旧系统数据 | 判断数据质量和迁移难度 | 基础资料重复、停用状态混乱 |
| 月结资料 | 梳理关账顺序和对账关系 | 月结往往依赖个人经验 |

### 角色安排

调研必须找对人。

| 角色 | 需要回答的问题 |
| --- | --- |
| 业务负责人 | 未来业务目标、流程取舍、审批原则 |
| 财务负责人 | 核算口径、账簿、科目、报表、月结 |
| 仓库负责人 | 收发存、盘点、批号、库位、条码 |
| 采购负责人 | 采购流程、供应商、价格、到票付款 |
| 销售负责人 | 销售订单、发货、开票、收款、信用 |
| 生产负责人 | BOM、计划、领料、完工、委外 |
| IT 负责人 | 环境、账号、接口、数据、权限、安全 |
| 关键用户 | 真实操作细节、例外流程、历史问题 |

## 三、调研方法

### 先问业务目标

不要一开始就问菜单和按钮。先明确客户为什么上系统。

常见目标：

- 财务希望业务单据自动生成凭证，减少手工录入。
- 仓库希望库存数量及时准确，减少账实不符。
- 采购希望订单、入库、发票、付款可追踪。
- 销售希望订单、发货、开票、收款一体化。
- 生产希望从销售需求到 MRP、生产订单、领料、完工闭环。
- 管理层希望报表按组织、部门、客户、产品、项目分析。

每个目标都要落到可验收指标，例如：

- 月结时间从 7 天压缩到 3 天。
- 库存准确率达到 98% 以上。
- 业务凭证自动生成比例达到 90% 以上。
- 采购订单到付款全链路可查询。

### 再画现状流程

现状流程要记录动作、单据、岗位、系统和例外。

```text
销售接单
  -> 销售订单
  -> 信用或价格审核
  -> 发货通知
  -> 仓库出库
  -> 财务开票
  -> 应收确认
  -> 收款核销
```

每个节点至少问清：

- 谁发起。
- 用什么单据。
- 审批人是谁。
- 数据从哪里来。
- 下游谁使用。
- 失败或退回怎么处理。
- 月末怎么对账。

### 用案例倒逼细节

泛泛问“销售流程是什么”很容易得到空话。更有效的方法是拿一笔真实业务走流程：

> 客户 A 在 6 月 3 日下单 100 件产品，要求分两批发货，第一批 60 件，第二批 40 件；第一批已经出库但未开票，第二批客户临时取消 10 件。系统里应该怎么体现？

这个案例可以同时验证：

- 销售订单是否允许变更。
- 出库是否允许部分出库。
- 开票是否按出库还是按订单。
- 应收确认时点。
- 取消数量怎么关闭。
- 报表如何统计未出库、未开票、未收款。

## 四、各模块调研重点

### 财务

| 主题 | 关键问题 | 蓝图输出 |
| --- | --- | --- |
| 核算组织 | 哪些法人或利润中心独立核算 | 组织和账簿方案 |
| 会计政策 | 准则、币别、期间、汇率 | 账簿参数和期间方案 |
| 科目和维度 | 是否按部门、客户、供应商、项目、产品核算 | 科目表和辅助核算矩阵 |
| 凭证来源 | 哪些凭证来自业务，哪些手工 | 凭证模板和业务凭证边界 |
| 应收应付 | 开票确认还是出入库确认 | 应收应付流程和对账方案 |
| 固定资产 | 卡片分类、折旧政策、资产调拨 | 资产类别和折旧方案 |
| 报表 | 法定报表、管理报表、老板报表 | 报表清单和取数口径 |
| 月结 | 模块关账顺序和责任人 | 月结流程表 |

### 供应链

| 主题 | 关键问题 | 蓝图输出 |
| --- | --- | --- |
| 采购 | 申请、订单、收料、入库、退料、发票、付款 | 采购流程和单据转换 |
| 销售 | 报价、订单、发货、退货、开票、收款 | 销售流程和信用价格控制 |
| 库存 | 仓库、库位、批号、序列号、保质期 | 库存维度和盘点方案 |
| 存货核算 | 计价方法、暂估、差异处理 | 计价和成本核算方案 |
| 跨组织 | 调拨、组织间结算、内部交易 | 跨组织流程和结算规则 |

### 生产制造

| 主题 | 关键问题 | 蓝图输出 |
| --- | --- | --- |
| 工程数据 | BOM、替代料、损耗、版本 | BOM 管理规则 |
| 计划 | MRP 范围、需求来源、安全库存 | 计划参数和运行规则 |
| 生产 | 生产订单、领料、补料、退料、完工 | 生产执行流程 |
| 委外 | 委外订单、发料、入库、结算 | 委外闭环流程 |
| 成本 | 材料、人工、制造费用、在制品 | 成本对象和分配规则 |

### 开发和集成

| 主题 | 关键问题 | 蓝图输出 |
| --- | --- | --- |
| 二开 | 标准功能是否满足，差异是否必须开发 | 二开清单和验收标准 |
| 接口 | 外部系统、方向、频率、失败补偿 | 接口清单和字段映射 |
| 数据 | 历史数据是否迁移，迁移到什么粒度 | 数据迁移范围 |
| 安全 | 权限、日志、审计、敏感字段 | 权限和审计方案 |

## 五、蓝图文档结构

一份可用的蓝图建议包含：

```text
1 项目背景和目标
2 实施范围
3 组织和基础资料方案
4 业务流程方案
5 财务核算和凭证方案
6 权限和审批方案
7 数据迁移方案
8 报表和查询方案
9 二开和接口方案
10 测试、培训和上线方案
11 风险、假设和待确认事项
12 附件：流程图、配置表、字段清单
```

## 六、蓝图评审方法

### 评审前

- 把蓝图提前发给关键用户。
- 标注待确认事项，不要藏在正文里。
- 准备端到端案例，而不是只讲 PPT。
- 把标准功能、配置、二开、暂不实现分清楚。

### 评审中

按流程走，不按部门吵。

示例评审顺序：

1. 采购到付款。
2. 销售到收款。
3. 生产到入库。
4. 库存盘点和调拨。
5. 业务生成凭证。
6. 月末结账。
7. 管理报表。

每个流程都要确认：

- 单据从哪里开始。
- 哪些字段必填。
- 哪些环节审批。
- 单据之间如何转换。
- 失败、退货、红冲、关闭怎么处理。
- 最终如何影响库存、应收应付、总账和报表。

### 评审后

- 输出会议纪要。
- 把争议点列成清单。
- 明确责任人和截止日期。
- 蓝图修改后重新发版。
- 关键页需要客户签字确认。

## 七、差异分析

差异不要只写“客户要求开发”。要判断是否真有必要。

| 差异类型 | 处理方式 |
| --- | --- |
| 标准功能已有 | 培训用户按标准流程操作 |
| 参数可配置 | 写入配置清单并测试 |
| 表单字段缺失 | 评估字段、权限、报表、接口影响 |
| 流程差异 | 先评估是否调整业务流程 |
| 报表差异 | 定义取数口径和展示格式 |
| 接口需求 | 写清字段、频率、失败处理和日志 |
| 必须二开 | 写需求说明、测试用例和验收标准 |

## 八、实际案例

### 案例：制造企业上线财务供应链制造一体化

背景：

- 客户原来用 Excel 管理 BOM 和库存。
- 财务用独立财务软件做凭证。
- 生产领料经常先拿料后补单，月末成本差异大。

调研发现：

- 物料编码重复，同一种螺丝有 5 个编码。
- BOM 版本没有生效日期。
- 仓库没有库位，批号只在纸面记录。
- 采购入库和发票跨月，暂估处理不稳定。
- 财务总账只看到月底汇总，无法追到业务单据。

蓝图方案：

- 先清理物料，建立统一编码和物料属性。
- BOM 设置版本、生效日期和审核流程。
- 库存启用批号，关键仓库启用库位。
- 采购入库生成暂估，应付发票到票后冲暂估。
- 生产订单按 BOM 领料，例外领料必须审批。
- 完工入库后进入存货核算，月末按成本规则分配费用。
- 业务单据通过凭证模板生成总账凭证。

验收案例：

1. 销售订单产生需求。
2. MRP 生成采购建议和生产建议。
3. 采购入库形成库存。
4. 生产订单领料并完工。
5. 销售出库并开票。
6. 应收收款核销。
7. 月末存货核算、成本计算、业务生成凭证、总账结账。

只有这条链路走通，蓝图才算真正可执行。

## 十、常见误区

| 误区 | 后果 | 正确做法 |
| --- | --- | --- |
| 只问操作，不问业务目的 | 系统上线后仍解决不了痛点 | 先问目标，再问流程 |
| 把旧流程照搬 | 低效流程被系统固化 | 借蓝图阶段优化流程 |
| 不让财务参与业务流程 | 凭证、成本、对账出问题 | 业务流程必须验证财务影响 |
| 蓝图不写例外流程 | 上线后异常单据无法处理 | 退货、红冲、关闭、差异都要写 |
| 二开需求不设验收标准 | 开发完成后争议不断 | 每个二开必须有测试用例 |
| 报表只给样式 | 上线后口径争议 | 每个字段写来源和计算方式 |

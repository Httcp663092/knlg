来源：金蝶开发者官网产品页中的“苍穹水平分库：驾驭海量数据的不二选择”。

## 功能定位

苍穹单据分表后，随着时间推移，水平分表数据仍可能在单数据库内持续膨胀，导致单库无法支撑业务使用。水平分库是在水平分表引擎基础上的升级，用于解决分表单据因时间维度累积导致的单库容量和性能问题。

## 适用对象

官方资料中列出的适用对象：

- 实施交付人员。
- 开发人员。
- 运维人员。
- 客户系统管理员。

## 适用版本

官方资料提到适用版本为：

```text
金蝶 AI 苍穹 V7.0.1 及以上
```

## 适用场景

| 场景 | 说明 |
| --- | --- |
| 单据历史数据量巨大 | 单表或分表后仍不断增长 |
| 数据按时间增长明显 | 业务单据按日期、财务期间持续累积 |
| 单库容量压力大 | 一个数据库难以承载长期数据 |
| 查询性能下降 | 历史数据拖慢当前业务查询 |
| 分表已不足够 | 需要从单库分表升级到多库分担 |

## 前提条件

官方资料中提到，分表单据需要满足条件才可以进行分库配置：

- 分片属性包含日期或财务期间属性。
- 分片策略支持按时间维度进行切分。

这意味着水平分库更适合按时间增长、按期间归档或按日期查询明显的单据。

## 水平分库与数据清理的关系

| 能力 | 解决问题 | 使用时机 |
| --- | --- | --- |
| 数据清理 | 过期或失效数据从当前库表清理 | 数据不需要保留在当前业务库 |
| 水平分表 | 单表过大拆成多表 | 单库还能支撑，但单表压力大 |
| 水平分库 | 单库压力过大拆到多库 | 单库容量或性能达到瓶颈 |

实际项目中通常先评估：

1. 能不能通过归档或清理减少数据。
2. 能不能通过分表解决单表压力。
3. 是否必须水平分库。

## 实施步骤建议

1. 确认单据是否已经分表。
2. 分析数据增长维度，是按日期、财务期间还是其他字段增长。
3. 确认分片属性是否包含日期或财务期间。
4. 评估单库容量、查询性能和未来数据增长。
5. 在测试环境配置分库方案。
6. 准备历史数据迁移或路由策略。
7. 验证新增、查询、列表、报表、接口、权限。
8. 制定上线窗口和回退方案。
9. 上线后监控各库数据量和查询性能。

## 风险点

- 分片字段选择错误会导致数据分布不均。
- 报表和 SQL 如果没有适配分库，可能查不全。
- 二开 SQL 直接查物理表时，可能绕过平台路由。
- 历史数据迁移要有校验口径。
- 备份、恢复、监控要从单库思路改成多库思路。

## 排查清单

### 查询变慢

检查：

- 查询条件是否包含分片字段。
- 是否跨多个分库扫描。
- 是否有二开 SQL 直连物理表。
- 分库数据是否倾斜。

### 数据查不全

检查：

- 查询是否经过平台标准服务。
- 报表是否支持分库路由。
- 二开插件是否写死数据库或表名。
- 历史数据是否迁移完整。

### 上线后数据写入异常

检查：

- 分片字段是否为空。
- 分片策略是否覆盖当前期间。
- 数据库连接和权限是否正确。
- 新增单据是否走标准保存逻辑。

## 与财务期间的关系

如果单据按财务期间分片，实施时要确认：

- 会计期间是否稳定。
- 跨期单据如何处理。
- 反结账或跨期调整是否影响路由。
- 月结后历史期间数据是否仍允许修改。
- 报表是否需要跨期间汇总。

## 二、用户、角色和权限的关系

金蝶权限体系可以理解为：

```text
用户
  -> 绑定人员
  -> 分配角色
  -> 角色拥有功能权限
  -> 用户再受组织和数据权限限制
```

不要只看菜单权限。一个用户能不能完成业务，要同时看：

- 用户是否启用。
- 是否绑定人员。
- 是否分配角色。
- 角色是否有菜单和按钮权限。
- 是否有组织范围。
- 是否有数据权限。
- 是否有字段权限。
- 是否是工作流审批人。

## 三、角色设计

官方角色资料强调角色是权限集合。项目里建议按岗位设计角色，而不是按个人设计角色。

### 推荐角色粒度

| 角色 | 示例权限 |
| --- | --- |
| 采购员 | 采购订单新增、提交、查询 |
| 采购经理 | 采购订单审核、价格查询 |
| 仓管员 | 入库、出库、盘点 |
| 应收会计 | 应收单、收款单、核销 |
| 总账会计 | 凭证、过账、结账 |
| 系统管理员 | 用户、角色、参数维护 |

### 不建议

- 一个用户一个角色。
- 所有财务共用一个大角色。
- 普通用户长期拥有管理员权限。
- 接口账号复用个人角色。

## 四、功能授权和按钮权限

功能授权不只是“能不能进菜单”，还包括：

- 新增。
- 保存。
- 提交。
- 审核。
- 反审核。
- 删除。
- 导入。
- 导出。
- 打印。
- 下推。

常见问题：

| 现象 | 可能原因 |
| --- | --- |
| 能进菜单但不能新增 | 没有新增按钮权限 |
| 能保存但不能提交 | 没有提交权限或流程限制 |
| 能审核但不能反审核 | 反审核权限未给或下游单据限制 |
| 能查不能导出 | 导出权限受限 |

## 五、数据权限

数据权限决定用户能看哪些数据。

常见控制维度：

- 组织。
- 部门。
- 仓库。
- 客户。
- 供应商。
- 业务员。
- 项目。
- 单据范围。

### 排查路径

用户看不到数据时，按顺序查：

1. 单据是否存在。
2. 当前组织是否正确。
3. 用户是否有菜单权限。
4. 用户是否有组织权限。
5. 用户是否有数据权限。
6. 列表过滤条件是否限制。
7. 是否被字段或数据规则过滤。

## 六、移动应用安全

移动端审批、报工、扫码、收付款查询越来越常见。移动安全要关注：

- 是否允许移动端访问。
- 是否限制设备。
- 是否限制登录方式。
- 是否启用手势、验证码或二次认证。
- 移动端能否查看敏感字段。
- 离职用户移动端是否同步禁用。
- 移动端附件和图片是否安全。

### 移动端风险

- 审批人手机丢失。
- 共享账号在移动端登录。
- 移动端越权查看财务数据。
- 离职用户仍能通过移动端访问。

## 七、业务监控和业务监控平台

官方业务监控资料可用于配置事后提醒和风险预警。项目里应把它当成“系统自动巡检能力”。

### 适用场景

| 场景 | 监控条件 |
| --- | --- |
| 采购订单超期 | 交货日期已过且未入库数量大于 0 |
| 销售订单超期 | 交货日期已过且未出库数量大于 0 |
| 应收逾期 | 到期日已过且未收金额大于 0 |
| 应付到期 | 到期日临近且未付款 |
| 库存低量 | 即时库存低于安全库存 |
| 未审核单据 | 单据创建超过 N 天仍未审核 |
| 接口失败 | 失败日志超过阈值 |
| 月结异常 | 未生成凭证或未过账凭证存在 |

## 八、监控配置思路

```text
选择监控对象
  -> 定义过滤条件
  -> 定义触发频率
  -> 定义提醒对象
  -> 定义消息方式
  -> 定义处理动作
  -> 记录关闭结果
```

监控不是越多越好。太多无效提醒会导致用户忽略真正风险。

## 九、权限和监控结合

业务监控要按责任人推送：

- 采购超期推采购员和采购经理。
- 应收逾期推应收会计和销售负责人。
- 未过账凭证推总账会计。
- 库存低量推计划员或仓库主管。
- 接口失败推系统管理员和接口负责人。

如果提醒对象配置成所有人，最终很可能没人处理。

## 十、运维检查清单

- [ ] 用户启用、禁用、离职流程明确。
- [ ] 角色按岗位设计。
- [ ] 菜单、按钮、数据、字段权限分层测试。
- [ ] 高危权限单独审批。
- [ ] 接口账号独立维护。
- [ ] 移动端访问和敏感字段已控制。
- [ ] 业务监控只配置关键风险。
- [ ] 每个监控有责任人和关闭标准。
- [ ] 权限和监控变更有记录。

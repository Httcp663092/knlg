这篇用于建立金蝶数据排查和 SQL 取数的基本方法。不同金蝶产品、版本、部署模式、数据库类型和二开都会影响表结构，生产环境绝不能凭经验直接写修改 SQL。

## 一、SQL 使用红线

### 可以做

- 查询数据。
- 对比前台报表和后台数据。
- 定位异常单据。
- 统计导入结果。
- 生成问题排查证据。
- 在测试环境验证修复方案。

### 不要做

- 未备份直接改生产数据。
- 跳过业务流程改审核状态。
- 直接改已过账凭证。
- 直接改核销、结账、库存余额。
- 不经客户确认批量修复。
- 用 SQL 绕过权限和审批。

## 二、数据排查基本流程

```text
明确问题
  -> 固定业务口径
  -> 前台复现
  -> 找单据编号
  -> 查主表和分录表
  -> 查状态和关联关系
  -> 查凭证或下游单据
  -> 和前台报表交叉验证
  -> 输出证据和处理建议
```

## 三、先固定业务口径

查数据前必须写清：

| 口径 | 示例 |
| --- | --- |
| 产品和版本 | 金蝶云·星空、金蝶云·苍穹 |
| 数据中心 | 生产、测试、UAT |
| 组织 | A 公司、B 工厂 |
| 账簿 | 主账簿、副账簿 |
| 期间 | 2026-06 |
| 单据类型 | 应收单、销售出库单 |
| 状态 | 暂存、提交、审核、关闭 |
| 金额口径 | 含税、不含税、本位币、原币 |

没有口径的 SQL 很容易查出“看似正确但业务错误”的结果。

## 四、常见数据结构思路

金蝶业务单据通常有这些层级：

```text
单据头
  -> 单据体分录
  -> 子分录或关联明细
  -> 审核状态
  -> 上下游关联
  -> 凭证或账务结果
```

常见字段类型：

| 字段 | 意义 |
| --- | --- |
| 单据编号 | 用户可见编号 |
| 内码/ID | 系统唯一标识 |
| 单据类型 | 区分业务类型 |
| 组织 | 业务或核算归属 |
| 业务日期 | 业务发生日期 |
| 创建人/审核人 | 操作留痕 |
| 单据状态 | 暂存、提交、审核等 |
| 关闭状态 | 是否关闭 |
| 金额字段 | 含税金额、未税金额、税额 |
| 数量字段 | 基本单位数量、业务单位数量 |
| 源单字段 | 上游单据关联 |

## 五、查询单据的一般模板

先从前台拿到单据编号，再查后台。下面是结构模板，不是固定表名：

```sql
-- 1. 先查单据头
select *
from 单据头表
where 单据编号 = '目标单号';

-- 2. 再查单据分录
select *
from 单据分录表
where 单据头ID = '目标内码';

-- 3. 查状态和操作记录
select 单据编号, 单据状态, 审核人, 审核日期
from 单据头表
where 单据编号 = '目标单号';
```

实际项目中要以对应产品的元数据、数据库字典或开发平台查询结果为准。

## 六、查询凭证的一般思路

查凭证不要只查凭证表，要同时确认源单。

```text
源单编号
  -> 源单内码
  -> 凭证生成记录
  -> 凭证头
  -> 凭证分录
  -> 科目和辅助核算
  -> 过账状态
```

检查点：

- 凭证是否存在。
- 凭证是否审核。
- 凭证是否过账。
- 凭证分录科目是否正确。
- 辅助核算是否完整。
- 凭证能否反查源单。

## 七、对账类 SQL 思路

### 应收对总账

目标不是直接写一条大 SQL，而是分三层查：

1. 应收模块余额。
2. 应收来源凭证。
3. 总账科目余额。

差异分类：

- 应收单未生成凭证。
- 凭证未过账。
- 手工凭证影响应收科目。
- 期初差异。
- 查询条件不一致。

### 库存对总账

查询层次：

1. 即时库存数量。
2. 库存明细账数量。
3. 存货核算金额。
4. 存货凭证。
5. 总账存货科目余额。

不要用库存数量表直接判断财务金额，数量和金额属于不同口径。

## 八、常见排查 SQL 类型

| 类型 | 目的 |
| --- | --- |
| 单据状态查询 | 找未审核、未关闭、反审核单据 |
| 关联关系查询 | 找上下游断链 |
| 重复数据查询 | 找重复客户、供应商、物料 |
| 金额异常查询 | 找负数、零金额、税额异常 |
| 维度缺失查询 | 找客户、供应商、部门、项目为空 |
| 凭证状态查询 | 找未审核、未过账凭证 |
| 接口日志查询 | 找失败请求和重试记录 |
| 权限数据查询 | 找用户、角色、组织范围 |

## 九、SQL 结果验证

每条 SQL 结果都要回前台验证：

- 用单据编号回前台打开。
- 用账表查询同一口径。
- 和客户提供的截图或报表对比。
- 抽样核对明细。
- 汇总数和前台汇总一致。

SQL 结果如果和前台不一致，优先怀疑查询口径不一致，而不是立刻判断系统数据错。

## 十、数据修复流程

如果必须修复数据：

1. 先在测试环境复现。
2. 备份涉及数据。
3. 明确修复范围。
4. 写修复 SQL 或修复程序。
5. 在测试环境执行。
6. 前台验证。
7. 客户审批。
8. 生产执行。
9. 记录执行日志。
10. 修复后再次对账。

生产修复记录必须包含：

- 问题编号。
- 修复原因。
- 影响表和字段。
- 修复前后数据。
- 执行人。
- 审批人。
- 执行时间。

## 十一、性能注意

- 查询生产库要限制时间范围。
- 不要无条件扫大表。
- 大查询先在测试环境验证。
- 只查需要字段，少用 `select *`。
- 加上组织、期间、单据类型过滤。
- 导出大数据量前先确认服务器压力。
